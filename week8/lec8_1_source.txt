[objectives]
By the end of this lecture, students will be able to:
1. Explain how Protocol v1's four components translate into agent instructions for deployment.
2. Identify what changes when a written protocol becomes an operational agent instruction set.
3. Iterate with an AI to tighten protocol language for deployment context.
4. Recognize the difference between conversational protocol language and operational agent language.
5. Reflect on one specific moment from the agent segment where deployed behavior diverged from written expectations.
[/objectives]

[narration]
For two weeks you have been writing protocols. Today the course names what you have been doing: you have been designing agents. The four components of a minimal viable protocol — identity, scope, planning rule, escalation — are the same four components of an agent's instruction set. What changes when you move from document to deployment is precision. Agent instructions must be operational, not just descriptive. Today you translate Protocol v1 into agent instructions, build your first custom agent, and observe what your language actually produces.

**AI Interaction Protocol**

In this session, you will receive blocks of content with the
following structural pattern:

```
### <BlockID>
----------
<content>
----------
###
```

Some blocks include an interaction-mode directive on a separate
line inside the dashed envelope:

```
### <BlockID>
----------
[ai_interaction=<mode>]
<content>
----------
###
```

Three modes determine your response:

  - **respond** — Engage substantively with the block content.
    Treat it as a prompt, question, or task that wants your
    analytical response. Default for S blocks (prompts), M OPEN
    blocks (multi-turn explorations), and X blocks (free
    exploration).

  - **acknowledge** — Confirm receipt with one brief sentence
    naming the block ID. Do not analyze, comment on, or expand on
    the content. The student is filing the block for the record,
    not asking for engagement. Default for R0 blocks (this opener),
    R blocks (reflections), M CLOSE blocks (closing markers), Q
    blocks (structured selections), and O blocks (ordering
    selections).

  - **archive** — Respond with only the block ID and nothing else.
    Used for batched archival pastes where even brief
    acknowledgment would clutter the thread.

If a block has no explicit `[ai_interaction=...]` directive, apply
the block-type default per the list above.

This protocol applies for the remainder of this session.

directive: Evaluate the quality of the student's protocol-to-agent translation work. narration: Students translate their Protocol v1 plus Week 7 candidate additions into agent instructions for deployment in HokieAI. Developing means the student pasted Protocol v1 into the AI's prompt without iteration, or accepted the AI's first translation suggestion without engagement. Strong means the student engaged substantively with what changes when language must be operational rather than conversational, asked the AI specific questions about deployment implications (e.g., "what happens when this scope rule encounters an edge case?"), and produced agent instructions that are recognizably tighter, more specific, and more executable than their original Protocol v1.
dimensions: translation_depth, operational_tightening, deployment_inquiry, agent_instruction_quality
