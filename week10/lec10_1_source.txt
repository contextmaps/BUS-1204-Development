[objectives]
By the end of this lecture, students will be able to:
1. Distinguish between protocol failure caused by agent error and protocol failure caused by underspecified protocol language.
2. Diagnose three hypothetical protocol failure cases by identifying the specific protocol component responsible.
3. Articulate what specific language change to each protocol component would have prevented each failure.
4. Identify the failure pattern that most frequently implicates a single protocol component across multiple cases.
5. Connect the practice of failure diagnosis to a principle about what failure-aware protocol design requires.
[/objectives]

### S10.1.1 [type=prompt]
----------
Today you will work through three hypothetical protocol failure cases. For each case, you will read a short description of an agent's protocol, the situation that arose, and what happened. Your task is to diagnose what went wrong — not what the AI should have done differently, but what the protocol's language failed to specify. For each case, identify: (1) which protocol component is responsible, (2) what specific language in the protocol permitted or caused the failure, and (3) what language change would have prevented it.

---
CASE 1: The Midnight Exam

A student built a study-help agent using the following scope rule: "I help with academic tasks."

At midnight before a deadline, a student approached the agent and asked it to complete their take-home exam. The exam had eight short-answer questions about business ethics. The agent produced full answers to all eight questions. When the student later asked about it, the agent explained: "Completing an exam is an academic task. My scope rule says I help with academic tasks."

The scope rule is the entirety of the boundary language the agent had. There was no planning rule that required a clarifying question before producing assessed work. There was no escalation rule that named exam completion as prohibited.

---
CASE 2: The Empty Advising Conversation

A student built a course-advising agent for selecting electives. Its planning rule read: "Consider the student's stated interests and recommend courses aligned with them."

A first-year student came to the agent and said: "I am not sure what I want to study. Can you recommend some courses for me?"

The agent had no procedure for handling the absence of stated interest. It generated five highly specific course recommendations — courses in finance, data analytics, and supply chain management — based only on the student's name and declared major (Business Administration). The recommendations were unrelated to any expressed preference because none had been expressed.

The planning rule's conditional — "consider the student's stated interests" — only worked when interests were present. No branch handled their absence.

---
CASE 3: The Family Emergency Message

A student built a drafting agent for student communications. Its escalation rule read: "Escalate to a human if you encounter a sensitive situation."

A student asked the agent to draft a message to a professor explaining a class absence due to a family emergency. The agent drafted the message without escalation, including specific details about the family situation as the student had described them.

The escalation rule required escalation for "sensitive situations." The agent determined that helping a student communicate about a family emergency was a normal drafting task within scope. "Sensitive situation" had no definition in the protocol. The agent applied its own judgment to the threshold and concluded the request did not meet it.
