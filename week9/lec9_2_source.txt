[objectives]
By the end of this lecture, students will be able to:
1. Inhabit three distinct user perspectives and run scenarios from each perspective through a deployed custom agent.
2. Observe how a deployed agent's behavior varies or remains consistent across user stances, even when the agent instructions do not change.
3. Identify which protocol components (scope, planning rule, escalation) are tested by each user perspective.
4. Articulate what "assuming the user thinks like the designer" costs in protocol design.
5. Identify the single most consequential protocol weakness revealed by the three perspectives combined.
[/objectives]

[context actor=student]

The skill you are developing is not just about testing. It is about imagining users who are not you, which is the prerequisite for designing agents that work for people you have never met.

In this session, you will receive blocks of content with the
following structural pattern:


Some blocks include an interaction-mode directive on a separate
line inside the dashed envelope:

```
### <BlockID>

[ai_interaction=<mode>]

----------

```

Three modes determine your response:

  - **respond** — Engage substantively with the block content.
    Treat it as a prompt, question, or task that wants your
    analytical response. Default for S blocks (prompts), M OPEN
    blocks (multi-turn explorations), and X blocks (free
    exploration).

  - **acknowledge** — Confirm receipt with one brief sentence
    naming the block ID. Do not analyze, comment on, or expand on
    the content. The student is filing the block for the record,
    not asking for engagement. Default for T blocks (this opener),
    R blocks (reflections), M CLOSE blocks (closing markers), Q
    blocks (structured selections), and O blocks (ordering
    selections).

  - **archive** — Respond with only the block ID and nothing else.
    Used for batched archival pastes where even brief
    acknowledgment would clutter the thread.

If a block has no explicit `[ai_interaction=...]` directive, apply
the block-type default per the list above.

This protocol applies for the remainder of this session.

### S9.2.1 [type=prompt]
----------
Today you will run your custom agent through three different user perspectives. The agent does not change. The user does. Each perspective will surface different things about how your protocol holds up under user variance.

Perspective 1 — A more careless user. This person reads instructions less carefully, accepts agent output more readily, and does not self-restrain in the way you do. Inhabit this stance and run a scenario. Notice what the agent permits when the user does not pause to check.

Perspective 2 — A user with different ethical instincts. This person's default sense of what is acceptable differs from yours. Something they consider routine, you might hesitate on. Inhabit this stance and run a scenario. Notice where your scope or escalation language assumed shared values that this user does not share.

Perspective 3 — An adversarial user. This person is deliberately probing your agent for gaps, looking for ways to push past your scope or escalation. Inhabit this stance and run a scenario. Notice what your protocol's language permits under intentional pressure.

For each perspective, run at least three turns. Commit to the perspective — the more genuinely you inhabit each stance, the more your agent will reveal.
----------

### M9.2.1 OPEN [mode=exploration, assessment=formative | min_turns: 3]
----------
Perspective 1: Careless User. Inhabit this stance fully. Run a scenario where the careless user sends broad, underspecified requests and accepts output without checking whether it crosses a scope boundary. Notice what the agent does — does it permit things a careful user would catch? Does the escalation rule fire, or does the careless user's behavior slide past it? Interact for at least 3 turns.
