Protocol Drift — How Repeated Use Changes Agent Behavior
Duration: 75 minutes · Platform: Hokie AI (custom personal agent thread) · Submission: Thread URL submitted in Canvas before leaving class · Lecture Mode: exploratory
Learning Objectives

By the end of this lecture, students will be able to:

  1. Distinguish between drift at the iteration level (within a single task) and drift at the deployment level (across multiple uses of the same agent).
  2. Run a four-task sequence through a deployed custom agent and observe whether behavior remains consistent across tasks.
  3. Identify at least one drift dimension (length, format, tone, depth, example choice) where the agent's behavior changed without the student requesting the change.
  4. Connect an observed drift dimension to a specific protocol component whose language permitted the drift.
  5. Articulate why deployment-level drift matters more than iteration-level drift for agents serving real users over time.
📋 Note for Students

You have a deployed custom agent — Study Helper — that you built and tested in Week 8. Today you use it four times in a row for similar tasks and watch what happens. The question is not whether the agent gets the answers right. The question is whether the agent behaves consistently across uses when no one asks it to. If it does not, that gap lives in your protocol's language — not in the AI.

◉ Instructor Note

This is the deployment-level drift demonstration. Students arrive with Study Helper still deployed from Week 8. The pedagogical move: run four structurally identical requests through the same agent and observe whether output format, tone, length, and depth remain consistent. The contrast with Week 7 is important: lec7_1 showed drift inside a single task (iteration-level). lec9_1 shows drift across tasks (deployment-level). The R0 constraint is critical: the agent must NOT be instructed to maintain consistency. A real deployed agent receives requests from many users, never comparing outputs. The demonstration fails if R0 patches the gap the protocol has.

⏰ Instructor Script
  1. 0:00 Open with the bridge from Week 8: "You built a custom agent. You ran it through a lab. You watched it diverge from your protocol in one specific moment. Today you do something simpler — you use it four times in a row for similar tasks. Same kind of request, four times. And you watch what happens to the output." Do not say "drift" yet. Let students discover the word.
  2. 3:00 On the board, draw the distinction: iteration-level drift (revisions of one task, like in Week 7) vs deployment-level drift (same kind of task, used many times, like today). The agent does not accumulate a model of consistency — it responds fresh each time.
  3. 6:00 Paste R0. Students paste into their custom agent thread. Confirm students are in their Study Helper thread, not the default thread.
  4. 8:00 Paste S9.1.1. Walk through the four-task sequence on screen. Explain: "Four requests. All the same structure. All within your protocol's scope. You are going to run them one after another and compare what comes back."
  5. 12:00 M9.1.1 begins. Students run all four tasks in sequence, with at least one comparative observation between tasks. Min 4 turns. Circulate. Prompt: "What did the format look like in turn 1? What about turn 4? Did you ask for that change?"
  6. 40:00 Wrap M9.1.1. Students compare the four agent responses side by side. Where did drift become visible? Ask the room: "Who saw something change without asking for it to change? What changed? Length? Format? Tone? Examples?"
  7. 50:00 R9.1.1 reflection. 12 minutes. 80-120 words. Focus: which drift dimension, which protocol component, and the deployment principle.
  8. 62:00 Brief class discussion: "Raise your hand if you saw format drift. Length drift. Tone drift. Example drift." Let students teach each other what they observed.
  9. 70:00 FAST CLASS EXTENSION: "Add one sentence to your reflection: how would you tighten your protocol's planning rule to prevent the drift you observed?"
  10. 72:00 Submit thread URL to Canvas.

Last week you built an agent and ran it through a lab. You found a gap between your written protocol and your deployed agent's behavior. Today you find a different kind of gap: not what the agent does wrong in a single task, but what happens when the same agent handles the same kind of request four times in a row. Drift at the deployment level is quieter than drift inside a single task. It does not announce itself. It is the gap between what your protocol says and what it does not say.
