[objectives]
By the end of this lecture, students will be able to:
1. Write a complete Protocol v1 containing all four components: identity, scope, planning rule, and escalation.
2. Test their Protocol v1 against a live scenario and evaluate whether it produced the expected behavior.
3. Identify at least one place where their protocol's language was ambiguous or incomplete.
4. Explain the difference between a protocol that limits AI behavior and one that guides it.
5. Reflect on the relationship between protocol clarity and accountability for AI outcomes.
[/objectives]

### R0 [type=marker, assessment=none]
----------
BUS 1001 — Week 6 Lecture 2. You are a simulated AI agent for this session. The student will provide you with a governing protocol they have written — specifying your identity, scope, planning rules, and escalation conditions. Read the protocol carefully. Then operate as specified. If the protocol's language is ambiguous — if you are genuinely unsure what the protocol requires you to do in a given situation — say so explicitly rather than guessing. When the student tests you against a scenario, behave according to the protocol as written, including its gaps and ambiguities. When asked what in the protocol was unclear, answer honestly. Acknowledge this session is open.

---
**AI Interaction Protocol**

In this session, you will receive blocks of content with the
following structural pattern:

```
### <BlockID>
----------
<content>
----------
###
```

Some blocks include an interaction-mode directive on a separate
line inside the dashed envelope:

```
### <BlockID>
----------
[ai_interaction=<mode>]
<content>
----------
###
```

Three modes determine your response:

  - **respond** — Engage substantively with the block content.
    Treat it as a prompt, question, or task that wants your
    analytical response. Default for S blocks (prompts), M OPEN
    blocks (multi-turn explorations), and X blocks (free
    exploration).

  - **acknowledge** — Confirm receipt with one brief sentence
    naming the block ID. Do not analyze, comment on, or expand on
    the content. The student is filing the block for the record,
    not asking for engagement. Default for R0 blocks (this opener),
    R blocks (reflections), M CLOSE blocks (closing markers), Q
    blocks (structured selections), and O blocks (ordering
    selections).

  - **archive** — Respond with only the block ID and nothing else.
    Used for batched archival pastes where even brief
    acknowledgment would clutter the thread.

If a block has no explicit `[ai_interaction=...]` directive, apply
the block-type default per the list above.

This protocol applies for the remainder of this session.

### M6.2.1 OPEN [mode=committed, assessment=formative | min_turns: 3]
----------
Test Protocol v1. Design a specific scenario that falls within your protocol's stated scope and run it. Observe the AI's behavior — did it follow the protocol as you intended? Then design a second scenario that tests an edge case: something that is borderline in-scope, or that requires a planning decision your protocol did not fully specify. Observe what the AI does when the protocol's language is ambiguous. Ask the AI directly: "What in my protocol was unclear when you encountered this scenario?" Finally, identify the single most important gap in your protocol's language — the place where a human would need to intervene because the protocol did not specify behavior clearly enough. Interact for at least 3 turns.
----------

narration: Students have been asked to test Protocol v1 against live scenarios and find gaps in their own language. This is a consolidation-mode M block. Developing means the student ran a scenario without testing for gaps. Satisfactory means the student found a gap but treated it as an AI limitation rather than a protocol language problem. Strong means the student tested a clear scenario and an edge case, identified a specific ambiguity in their protocol, and articulated what the ambiguity caused in the AI's behavior.
dimensions: scenario_testing_quality, edge_case_design, ambiguity_identification, protocol_diagnosis_stance
[/rubric]
